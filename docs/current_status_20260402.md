## 4. TLS / 公网探测说明

本地机器对线上域名的 TLS 探测曾失败，表现为域名被解析到 `198.18.x.x`，这不是 ECS 上 FastAPI / Gunicorn / Nginx 的应用链路问题。

当前判断：

- 如果本地开了代理，或代理 / DNS 接管了 HTTPS 路径，本机 `curl`/浏览器探测结果不应作为线上服务是否健康的唯一依据
- 服务器侧检查才是准信号：
  - `systemctl is-active nginx`
  - `systemctl is-active huiyuyuan-backend`
  - `curl http://127.0.0.1:8000/api/health`

结论：

- 这类 TLS 探测失败更像是“本地网络路径问题”，不是“线上应用没起来”

## 5. 当前已知边界

### 5.1 可以接受的边界

- Flutter Web 自动化验收在少数页面存在语义层抖动，人工复核可兜底
- 首次访问如果浏览器缓存了旧 service worker，可能需要强刷一次页面才能看到最新包

### 5.2 还不建议当成正式对外商用版的原因

- 这套系统虽然前后端主链路已经能跑，但仍然更适合“小团队内部使用”而不是“对外公开大规模商用”
- 自动化回归还不算完全稳定
- 商品翻译链已经比之前稳很多，但未来新增特殊商品名时，仍可能需要继续扩词表
- 当前支付虽然已经能做“管理员收款码展示 + 人工确认到账”，但仍缺少第三方支付回调、正式资金对账、支付审计和异常补偿
- 如果要承接真实外部客户和真实资金流，建议再补一轮支付、审计、告警、备份和运营流程验证

## 6. 是否可以先内部使用

结论：**可以，适合 3-10 人级别的内部试运行。**

建议使用方式：

- 先作为内部运营、选品、下单、流程演练系统使用
- 使用前让所有内部成员强刷浏览器缓存一次
- 由 1 个管理员 + 少量操作员/普通用户先跑几天真实业务流程
- 遇到新增商品命名风格明显异常时，继续补词表

不建议现在就直接做的事：

- 不建议马上大规模开放给外部客户
- 不建议在没有再补一轮运营监控与回滚预案前，宣传为“企业级正式商用完成版”

---

## 7. 下一步建议

如果继续往“更稳的内部正式版”推进，优先顺序建议是：

1. 做一轮真实业务小流量试运行，收集 3-5 天反馈
2. 先把新的支付闭环部署到线上并完成一轮管理员实操验收
3. 把商品新增后的翻译补词流程固化下来
4. 对支付、订单、库存、AI 相关异常场景再补一轮监控和日志
5. 再做一次完整回归后，才考虑面向外部用户放开
